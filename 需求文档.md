# 酒店周边竞品房价监控 + 临时热点活动采集系统
## 需求文档（PRD格式）

---

## 一、产品概述

### 1.1 产品愿景

打造一款面向酒店运营者的智能监控分析系统，通过实时采集竞品房价数据和周边临时热点活动，帮助酒店动态调整定价策略、优化房态管理，实现收益最大化。

### 1.2 目标用户

- **酒店运营经理**：需要实时掌握竞品动态和周边活动信息
- **收益管理团队**：需要数据分析支持定价决策
- **酒店业主/店长**：需要及时获取市场变化通知

### 1.3 解决的核心痛点

| 痛点 | 解决方案 |
|------|----------|
| 竞品调价无法及时知晓 | 实时采集多平台房价，分钟级监控 |
| 错过周边展会/考试高峰 | 自动采集周边3公里临时热点活动 |
| 定价决策缺乏数据支撑 | 可视化驾驶舱 + AI分析报告 |
| 信息获取不及时 | 微信定时/实时推送 |

## 二、功能需求

### 2.1 模块一：竞品房价监控采集

#### P0级功能（必须实现）

| 功能点 | 详细描述 |
|--------|----------|
| 多平台数据采集 | 支持美团、携程、飞猪三大OTA平台 |
| 竞品酒店配置 | 用户可自定义添加/删除竞品酒店列表 |
| 房型房价采集 | 采集各竞品酒店各房型（大床/双床/套房等）的实时房价 |
| 数据更新频率 | 支持配置采集频率（最低1次/小时） |
| 历史数据存储 | 保留至少30天历史数据，支持趋势分析 |

#### P1级功能（重要功能）

| 功能点 | 详细描述 |
|--------|----------|
| 异常波动告警 | 竞品房价异常波动（如单次调价超过20%）时触发告警 |
| 满房/低价预警 | 竞品出现满房或异常低价时推送通知 |
| 手动触发采集 | 支持手动刷新指定酒店/平台数据 |

### 2.2 模块二：周边临时热点活动采集

#### P0级功能（必须实现）

| 功能点 | 详细描述 |
|--------|----------|
| 活动类型覆盖 | 采集考试（考研/公考/CPA等）、展会（行业展/车展）、演唱会、体育赛事、节庆活动等 |
| 地理范围 | 默认周边3公里，支持自定义范围（1-10公里） |
| 时间维度 | 采集未来7-30天内的临时性活动 |
| 活动信息字段 | 活动名称、时间、地点、预计参与人数、活动类型标签 |
| 需求热度预估 | 根据活动类型和规模，自动评估对周边住宿需求的拉动程度（高/中/低） |

#### P1级功能（重要功能）

| 功能点 | 详细描述 |
|--------|----------|
| 多源数据采集 | 本地生活平台（大麦/活动行）、政府公告、学校官网、展会官网等 |
| 活动日历视图 | 以日历形式展示周边活动分布 |
| 历史活动分析 | 分析往期同类活动对房价的影响规律 |

### 2.3 模块三：可视化驾驶舱 & 分析功能

#### P0级功能（必须实现）

| 功能点 | 详细描述 |
|--------|----------|
| 竞品房价仪表盘 | 实时展示竞品房价对比，支持按酒店/房型/平台筛选 |
| 价格趋势图 | 折线图展示指定时间段内各竞品房价变化趋势 |
| 市场分布图 | 各酒店价格带分布、平台价格差异对比 |
| 热点活动地图 | 在地图上标注周边活动位置和热度 |

#### P1级功能（重要功能）

| 功能点 | 详细描述 |
|--------|----------|
| 定价策略分析 | 基于竞品价格和活动热度，提供定价建议（如"建议当前价格上调15%"） |
| 市场趋势分析 | 识别价格周期、周末/工作日差异、节假日效应 |
| 差异化策略分析 | 分析自身与竞品的价格差异、房型优势，输出差异化建议 |
| 报告导出 | 支持导出PDF/Excel格式的分析报告 |

### 2.4 模块四：大模型对接

| 功能点 | 详细描述 |
|--------|----------|
| 默认模型 | DeepSeek API对接（首选） |
| 模型切换 | 支持后续切换或增加其他大模型（如通义千问、GPT等） |
| AI分析能力 | - 自动生成每日/每周市场分析报告<br>- 基于活动热度预测需求高峰<br>- 智能定价建议生成 |

### 2.5 模块五：通知功能

| 功能点 | 详细描述 |
|--------|----------|
| 通知方式 | 微信（企业微信/个人微信，通过Server酱或WxPusher实现） |
| 定时推送 | 每日/每周定时推送采集数据汇总和分析报告 |
| 实时推送 | 满足触发条件时立即推送（如竞品调价、新活动出现） |
| 推送内容 | 竞品调价信息、新活动提醒、异常预警、分析报告摘要 |

## 四、技术架构建议

### 4.1 系统架构
```
┌─────────────────────────────────────────────────────────────┐
│                   用户界面层 (Web/移动端)                    │
├─────────────────────────────────────────────────────────────┤
│                   业务逻辑层 (API服务)                       │
├─────────────────────────────────────────────────────────────┤
│                   数据处理层 (数据采集/分析)                  │
├─────────────────────────────────────────────────────────────┤
│                   数据存储层 (数据库/缓存)                    │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 技术栈建议
- **后端框架**: Python FastAPI / Node.js NestJS
- **数据库**: PostgreSQL (主数据) + Redis (缓存)
- **数据采集**: Playwright / Puppeteer (网页爬虫)
- **可视化**: ECharts / Ant Design Charts
- **消息推送**: Server酱 / WxPusher API
- **部署**: Docker + Kubernetes

### 4.3 数据流设计
1. **定时任务** → 采集OTA平台房价数据
2. **活动爬虫** → 采集周边活动信息
3. **数据处理** → 清洗、分析、存储
4. **AI分析** → 大模型生成报告和建议
5. **通知系统** → 推送结果给用户

## 七、数据字段定义

### 7.1 房价数据字段
| 字段 | 类型 | 说明 |
|------|------|------|

## 九、风险与应对

### 9.1 技术风险
- **风险**: OTA平台反爬虫机制
- **应对**: 使用代理IP池、模拟真人操作、遵守robots.txt

### 9.2 数据风险
- **风险**: 数据采集不准确或不完整
- **应对**: 多源数据验证、人工审核机制、数据质量监控

### 9.3 法律风险
- **风险**: 数据采集可能涉及隐私或版权问题
- **应对**: 仅采集公开数据、设置采集频率限制、提供数据删除功能

### 11.2 技术栈建议
- **前端框架**: React 18 + TypeScript
- **UI组件库**: Ant Design / Element Plus
- **图表库**: ECharts / Ant Design Charts
- **地图**: 高德地图API / 百度地图API
- **后端框架**: Node.js NestJS / Python FastAPI
- **数据库**: PostgreSQL + Redis
- **数据采集**: Playwright + Puppeteer
- **消息队列**: RabbitMQ / Redis Stream

- **消息推送**: Server酱 / WxPusher / 企业微信API

5. **数据存储** → 写入数据库，更新缓存
6. **AI分析** → 调用DeepSeek API生成分析报告
