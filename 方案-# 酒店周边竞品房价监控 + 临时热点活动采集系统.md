# 需求分析文档

## 1. 需求概述
为酒店运营者提供竞品房价与周边活动的实时监控与分析系统。通过 Chrome Extension 辅助采集美团商家后台数据，结合活动信息自动采集，实现数据可视化与AI智能分析。

## 2. 目标用户
- **核心用户（酒店运营经理/收益管理专员）**：需要实时掌握市场动态以调整定价策略的中小型酒店或连锁酒店单店运营人员。特征：关注收益，决策压力大，时间碎片化。场景：每日上班后、重大活动前、竞品突然调价时查看系统。
- **次要用户（酒店业主/店长）**：需要宏观了解市场趋势和经营建议的决策者。特征：关注整体收益和报告，不频繁操作。场景：每周或每月查看分析报告，接收重大预警。

## 4. 数据采集方案

### 4.1 房价数据采集：Chrome Extension方案

**核心思路**：不代登录、不存密码，用户自己登录美团商家后台，Chrome Extension 辅助提取页面数据并上报。

**数据来源**（美团商家后台）：
| 数据类型 | 说明 | 采集方式 |
|---------|------|---------|
| 竞争圈数据 | 同商圈/同品类竞品的价格、销量、评分 | Extension提取商家后台"竞争圈"页面DOM |
| 行业对标数据 | 同城竞品的经营指标对比 | Extension提取"行业对标"页面数据 |
| 经营数据报表 | 订单量、营业额、客单价、流量转化 | Extension提取"经营报表"页面数据 |

**工作流程**：
```
1. 用户安装 Chrome Extension → 扫码/登录绑定系统账号
2. 用户正常访问美团商家后台（使用自己的账号登录）
3. Extension 检测到商家后台页面 → 自动提取竞争圈/行业对标/经营数据
4. Extension 将数据通过 HTTPS API 上报至后端
5. 后端存储并触发告警检测
```

**合规性**：
- 不存储用户的美团账号密码
- 不代登录，用户自行访问自己的商家后台
- Extension 仅读取用户自己能看到的数据
- 用户可随时在 Extension 中暂停/启用采集

## 5. 用户场景
### 场景1: 应对竞品突然降价
**用户**: 酒店运营经理张经理
**目标**: 及时发现并应对主要竞品的降价行为，避免客源流失。
**步骤**:
1. 张经理在系统中设置了A、B两家酒店为竞品，并设定了"价格降幅超15%"的告警规则。
2. 张经理打开Chrome浏览器访问美团商家后台，Extension自动采集竞争圈数据。
3. 系统检测到A酒店的豪华大床房价格下调20%。
4. 系统立即通过微信向张经理发送一条包含降价详情和对比数据的告警消息。
5. 张经理点击消息中的链接，跳转到系统驾驶舱，查看详细的价格趋势图和自身房态。
6. 系统AI分析模块基于此降价和周边无大型活动的情况，给出了"建议保持现价观察，或推出限时权益包"的建议。
**预期结果**: 张经理在5分钟内获知市场变化，并基于数据支持做出了稳妥的应对决策，稳住了当期收益。

### 场景2: 提前布局大型活动期间定价
**用户**: 收益管理专员李专员
**目标**: 为下个月的大型展会提前制定有竞争力的房价策略。
**步骤**:
1. 系统活动采集模块从中国展会网抓取到"国际车展"将于30天后在距酒店2公里的会展中心举办，预计人流5万。
2. 系统自动评估该活动为"高"需求热度，并将其标注在活动日历和地图上。
3. 李专员每周一查看系统推送的《周边活动周报》时，发现了该高热度活动。
4. 她进入系统，筛选查看车展期间（前后共5天）的历史同期房价数据，以及主要竞品在当前可预订期的价格。
5. 她使用系统的"定价模拟"功能（P1），输入预期涨幅，系统结合竞品价和活动热度给出可行性分析和收益预估。
6. 李专员根据分析结果，制定了分阶段的预售价格策略，并提前上架了相关套餐。
**预期结果**: 酒店在需求高峰来临前完成了价格部署，抓住了市场机会，实现了收益最大化。

## 6. 非功能需求
- **性能要求**: 核心数据驾驶舱页面加载时间<3秒；Extension单次数据采集与上报完成时间<10秒；活动采集任务完成时间<5分钟；支持单用户同时监控100+竞品酒店。
- **安全要求**: 用户密码加密存储；不存储用户美团账号密码；Extension通信全程HTTPS；支持基于角色的访问控制（如运营经理可配置，店员仅可查看）；采集数据仅用于内部分析，不外泄。
- **可用性要求**: 核心服务可用性目标≥99.5%；系统界面简洁，主要操作3步内完成，支持移动端适配查看；Chrome Extension安装后无需额外配置即可使用。

## 7. 产品边界
1. **不提供自动定价与房态同步**：本系统仅提供数据监控、分析与建议，不直接与酒店PMS（物业管理系统）或OTA后台对接以自动修改价格或房态。
2. **不替代人工决策**：所有AI生成的分析与建议仅供参考，最终决策需由酒店运营人员结合实际情况做出。
3. **不代登录商家后台**：Chrome Extension仅在用户自己登录商家后台后辅助提取数据，不存储账号密码，不代替用户登录。
4. **MVP仅覆盖美团平台**：首期仅支持美团商家后台数据采集及主流活动发布渠道，后续视情况扩展携程、飞猪等平台。
5. **活动采集仅采集公开数据**：所有活动信息采集均来自公开网站，不涉及破解或获取非公开数据。

# 架构设计文档

## 1. 技术选型
| 层面 | 技术方案 | 选型理由 |
|-----|---------|---------|
| 前端 | Vue 3 + TypeScript + Vite + Element Plus + ECharts | Vue 3生态成熟，TypeScript提升代码健壮性，Vite构建速度快，Element Plus组件库丰富，ECharts满足复杂图表需求。支持响应式布局，适配移动端查看。 |
| 浏览器扩展 | Chrome Extension (Manifest V3) | 用户已登录美团商家后台，Extension辅助提取页面数据并上报。Manifest V3是Chrome扩展最新标准，安全性与性能更优。 |
| 后端 | Python (FastAPI) + Celery + Redis | Python在数据分析和AI集成方面生态强大。FastAPI高性能，异步支持好，自动生成API文档。Celery用于管理异步任务（活动采集、告警检测、报告生成），Redis作为消息代理和缓存。 |
| 数据库 | PostgreSQL + TimescaleDB (时序扩展) | PostgreSQL功能强大，可靠性高，支持JSON、GIS等复杂类型。TimescaleDB作为PostgreSQL扩展，专为高效存储和查询时序数据（如房价）优化。两者在同一数据库实例中运行。 |
| 部署 | Docker + Docker Compose | 容器化确保环境一致性，简化依赖管理。Docker Compose适合MVP阶段的快速部署和开发。 |

## 2. 系统架构
```
整体采用前后端分离架构，核心分为展示层、应用层、数据层和基础设施层。

[用户] -> (微信/浏览器) -> [Nginx反向代理] -> [前端静态资源]
                                      |
                                      v
                              [后端API服务 (FastAPI)]
                                      |
        +-----------------------------+-----------------------------+
        |                             |                             |
        v                             v                             v
[业务逻辑服务]              [异步任务引擎 (Celery)]        [AI分析服务]
  - 用户管理                  - 调度器 (Celery Beat)         - 集成大模型API
  - 竞品/活动配置             - 活动采集Worker               - 报告生成
  - 告警规则处理              - 数据分析Worker               - 定价建议模拟
  - 数据查询聚合              - 通知推送Worker
        |                             |
        v                             v
[数据库 (PostgreSQL + TimescaleDB)]  [缓存/消息队列 (Redis)]
  - 用户、配置、元数据               - 任务队列、状态缓存
  - 房价时序数据 (TimescaleDB)       - Extension在线状态
  - 活动数据、GIS数据

[用户浏览器]
  ├── 美团商家后台页面
  └── Chrome Extension
          │ (检测商家后台页面，提取数据)
          │ (HTTPS API上报数据)
          v
  [后端API服务 /api/v1/extension/*]

数据流：
1. Extension采集：用户访问美团商家后台 → Extension自动提取竞争圈/经营数据 → HTTPS上报后端 → 存入DB
2. 活动采集：Celery Beat定时触发活动采集任务 → Worker从各公开网站采集数据 → 清洗后存入DB
3. 告警分析：数据分析Worker周期性扫描新数据 → 比对告警规则 → 触发则调用通知Worker
4. 报告生成：AI分析服务定时或按需从DB聚合数据 → 调用大模型API生成报告
5. 前端展示：用户请求仪表盘 → 后端从DB聚合数据 → 通过API返回 → 前端渲染图表
6. 消息推送：通知Worker通过微信推送服务（Server酱/WxPusher）向用户发送消息
```

## 3. 模块划分
### 模块1: Chrome Extension数据采集模块
- **职责**: 以浏览器扩展形式运行，检测用户访问的美团商家后台页面，自动提取竞争圈数据、行业对标数据、经营报表数据，并通过API上报至后端。管理Extension与系统账号的绑定关系。
- **依赖**: 用户已登录美团商家后台（非系统控制）、后端API服务（数据上报）、用户系统账号（绑定认证）。
- **接口**: Extension内部逻辑，对外与后端通过 `POST /api/v1/extension/report`（数据上报）、`POST /api/v1/extension/register`（绑定账号）通信。

### 模块2: 活动数据采集模块
- **职责**: 负责从公开网站（教育考试网、展会网、文旅局官网等）定时采集活动信息。管理采集任务的创建、调度、执行和状态监控。
- **依赖**: 配置管理模块（获取采集目标）、Redis（任务队列）、数据存储模块。
- **接口**: `POST /api/v1/tasks/crawl/activity` (手动触发活动采集)， `GET /api/v1/tasks/status` (查询任务状态)。内部与Celery Worker通过消息队列通信。

### 模块3: 核心数据服务模块
- **职责**: 提供用户管理、竞品酒店管理、活动信息管理、告警规则配置等核心业务逻辑的CRUD操作和业务验证。
- **依赖**: 数据存储模块、用户认证模块。
- **接口**: `GET/POST/PUT/DELETE /api/v1/competitors`, `GET/POST/PUT/DELETE /api/v1/alert-rules`, `GET /api/v1/activities` 等Restful API。

### 模块4: 数据分析与告警模块
- **职责**: 处理时序数据，进行价格波动计算、活动热度评估，并根据用户配置的规则触发实时告警。
- **依赖**: 数据存储模块（读取时序数据）、通知推送模块（发送告警）。
- **接口**: 内部模块，通过事件或直接函数调用触发。对外提供 `POST /api/v1/analysis/price-trend` (查询价格趋势) 等查询接口。

### 模块5: AI分析与报告生成模块
- **职责**: 聚合多维度数据，通过Prompt工程调用大模型（如DeepSeek）API，生成结构化市场分析报告和定价建议。
- **依赖**: 核心数据服务模块、数据分析模块（获取聚合数据）、大模型外部API。
- **接口**: `POST /api/v1/ai/generate-daily-report`, `POST /api/v1/ai/simulate-pricing` (定价模拟)。

### 模块6: 通知与推送模块
- **职责**: 统一管理所有对外通知渠道（微信），封装第三方推送服务（Server酱/WxPusher）的调用，实现消息模板化和发送记录。
- **依赖**: 无直接内部依赖，被其他模块（如告警模块、报告生成模块）调用。
- **接口**: 内部服务类，提供 `send_wechat_message(user_id, title, content, template_type)` 方法。

### 2.1 Content Script - 美团商家后台数据提取
```javascript
// content-scripts/meituan-merchant.js
// 检测当前页面是否为美团商家后台，自动提取数据

const MEITIAN_MERCHANT_PATTERNS = [
  '*://e.meituan.com/*',
  '*://kd.meituan.com/*'
];

// 竞争圈数据提取
function extractCompetitorData(selectors) {
  const competitors = [];
  document.querySelectorAll(selectors.competitorItem).forEach(item => {
    competitors.push({
      name: item.querySelector(selectors.hotelName)?.textContent?.trim(),
      price: parsePrice(item.querySelector(selectors.price)?.textContent),
      rating: item.querySelector(selectors.rating)?.textContent?.trim(),
      sales: item.querySelector(selectors.sales)?.textContent?.trim(),
    });
  });
  return competitors;
}

// 经营数据提取
function extractBusinessData(selectors) {
  return {
    orderCount: parseNumber(document.querySelector(selectors.orderCount)?.textContent),
    revenue: parsePrice(document.querySelector(selectors.revenue)?.textContent),
    avgPrice: parsePrice(document.querySelector(selectors.avgPrice)?.textContent),
    conversionRate: parsePercent(document.querySelector(selectors.conversionRate)?.textContent),
  };
}

// 数据采集主流程
async function collectPageData() {
  const pageType = detectPageType(); // 识别当前页面类型（竞争圈/经营报表/行业对标）
  const selectors = await getSelectors(); // 从本地/远程获取选择器配置

  let data = {};
  switch (pageType) {
    case 'competitor':
      data = extractCompetitorData(selectors);
      break;
    case 'business':
      data = extractBusinessData(selectors);
      break;
    case 'benchmark':
      data = extractBenchmarkData(selectors);
      break;
  }

  if (Object.keys(data).length > 0) {
    await reportToBackend(pageType, data);
  }
}

// 页面加载完成后自动采集
if (matchPattern(MEITIAN_MERCHANT_PATTERNS)) {
  // 页面加载完成后延迟采集，确保数据渲染完成
  setTimeout(collectPageData, 2000);
  // 监听页面变化（SPA路由切换）
  observePageChange(collectPageData);
}
```

### 2.2 选择器远程更新机制
```javascript
// Extension启动时从后端拉取最新的选择器配置
async function getSelectors() {
  const cached = await chrome.storage.local.get('meituan_selectors');
  const cacheTime = await chrome.storage.local.get('selectors_updated_at');
  const ONE_HOUR = 60 * 60 * 1000;

  // 缓存超过1小时则远程更新
  if (!cached || Date.now() - cacheTime > ONE_HOUR) {
    const remote = await fetchSelectorsFromBackend();
    await chrome.storage.local.set({
      meituan_selectors: remote,
      selectors_updated_at: Date.now()
    });
    return remote;
  }
  return cached;
}
```

### 2.3 数据上报
```javascript
// utils/api.js
async function reportToBackend(pageType, data) {
  const token = await getExtensionToken();
  const response = await fetch('https://api.our-system.com/api/v1/extension/report', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'X-Extension-Token': token
    },
    body: JSON.stringify({
      type: pageType,
      source: 'meituan_merchant',
      data: data,
      url: window.location.href,
      captured_at: new Date().toISOString()
    })
  });

  if (!response.ok) {
    // 上报失败，存入本地队列，稍后重试
    await storeForRetry(pageType, data);
  }
}
```

## 3. Extension安全设计
- **不存储美团账号密码**：Extension仅在用户已登录的页面上读取数据
- **Extension Token认证**：每个Extension实例注册时生成唯一Token，上报数据时携带
- **最小权限原则**：仅申请必要的Chrome权限（`storage`、`tabs`、指定域名`host_permissions`）
- **数据本地缓存**：上报失败时本地暂存，网络恢复后重试，不丢失数据
- **用户可控**：Popup面板提供开关，用户可随时暂停/启用数据采集

## 5. 关键交互流程
### 流程: Extension数据采集与告警
1. **用户触发**: 用户在浏览器中打开美团商家后台
2. **数据采集**: Extension自动检测页面类型，提取竞争圈/经营数据
3. **数据上报**: Extension通过HTTPS将数据上报至后端
4. **后端处理**:
   - 后端验证Extension Token，存储数据至TimescaleDB
   - 立即触发告警检测：比对新数据与用户设定的告警规则
   - 若触发告警，生成告警记录并推送微信通知
5. **界面响应**:
   - WebSocket推送告警到前端
   - 前端显示实时通知弹窗
   - 用户点击通知跳转到竞品详情页

### 流程: Extension安装与绑定
1. **用户触发**: 用户在系统中点击"安装Extension"
2. **安装引导**:
   - 显示安装步骤和下载链接
   - 引导用户安装Chrome Extension
3. **账号绑定**:
   - Extension安装后，用户在Extension Popup中扫码或输入系统账号登录
   - Extension生成设备ID，注册至后端
   - 后端返回Extension Token
4. **状态同步**:
   - 前端Extension状态页面显示"已连接"
   - 用户可查看Extension采集状态

- **响应格式**:

  ```json
    "message": "success",

        "id": "uuid",
        "username": "string",
        "role": "admin|manager|viewer"

    }
  ```

  1. 验证用户名密码

  4. 返回token和用户信息

- **方法**: POST

## 3. 数据库表结构

```sql
-- 启用必要扩展
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS timescaledb;

-- 用户表
CREATE TABLE users (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(255) UNIQUE NOT NULL,
    hashed_password VARCHAR(255) NOT NULL,
    full_name VARCHAR(100),
    role VARCHAR(20) NOT NULL DEFAULT 'viewer', -- admin, manager, viewer
    hotel_name VARCHAR(255),
    hotel_location GEOGRAPHY(Point),
    notification_preferences JSONB DEFAULT '{"wechat": true, "email": false}',
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 竞品酒店表
CREATE TABLE competitor_hotels (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    platform VARCHAR(50) NOT NULL CHECK (platform IN ('meituan', 'ctrip', 'fliggy')),
    external_id VARCHAR(255) NOT NULL,
    room_types JSONB DEFAULT '[]',
    location GEOGRAPHY(Point),
    is_active BOOLEAN DEFAULT true,
    last_collected_at TIMESTAMPTZ,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(user_id, platform, external_id)
);

-- 房价时序数据表 (TimescaleDB)
CREATE TABLE price_time_series (
    time TIMESTAMPTZ NOT NULL,
    competitor_hotel_id UUID NOT NULL REFERENCES competitor_hotels(id) ON DELETE CASCADE,
    room_type VARCHAR(100) NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    availability BOOLEAN,
    original_price DECIMAL(10,2),
    discount_rate DECIMAL(5,2),
    data_source VARCHAR(20) NOT NULL DEFAULT 'extension', -- extension, manual
    collected_at TIMESTAMPTZ NOT NULL,
    metadata JSONB DEFAULT '{}',
    PRIMARY KEY (time, competitor_hotel_id, room_type)
);

-- 创建时序超表
SELECT create_hypertable('price_time_series', 'time');

-- 周边活动表
CREATE TABLE surrounding_activities (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title VARCHAR(500) NOT NULL,
    description TEXT,
    start_time TIMESTAMPTZ NOT NULL,
    end_time TIMESTAMPTZ NOT NULL,
    location GEOGRAPHY(Point),
    address VARCHAR(500),
    source VARCHAR(100) NOT NULL,
    source_url VARCHAR(500),
    activity_type VARCHAR(50) NOT NULL CHECK (activity_type IN ('exam', 'exhibition', 'concert', 'other')),
    estimated_attendance INTEGER,
    demand_level VARCHAR(20) NOT NULL CHECK (demand_level IN ('HIGH', 'MEDIUM', 'LOW')),
    demand_score DECIMAL(3,2), -- 热度评分 0.00-1.00
    external_id VARCHAR(255), -- 来源网站上的唯一ID，用于去重
    collected_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(source, external_id)
);

-- 创建活动空间索引
CREATE INDEX idx_activities_location ON surrounding_activities USING GIST(location);
CREATE INDEX idx_activities_time ON surrounding_activities(start_time, end_time);
CREATE INDEX idx_activities_type ON surrounding_activities(activity_type);

-- 告警规则表
CREATE TABLE alert_rules (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name VARCHAR(255) NOT NULL,
    rule_type VARCHAR(50) NOT NULL CHECK (rule_type IN ('price_drop', 'price_rise', 'sold_out', 'low_price', 'availability_change')),
    competitor_ids JSONB NOT NULL DEFAULT '[]', -- 关联的竞品酒店ID列表
    conditions JSONB NOT NULL DEFAULT '{}', -- 规则条件，如 {"threshold": 15.0, "time_window": "1h", "room_types": ["豪华大床房"]}
    notification_channels JSONB NOT NULL DEFAULT '["wechat"]',
    is_active BOOLEAN DEFAULT true,
    last_triggered_at TIMESTAMPTZ,
    trigger_count INTEGER DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

-- 告警记录表
CREATE TABLE alert_records (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    alert_rule_id UUID NOT NULL REFERENCES alert_rules(id) ON DELETE CASCADE,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    competitor_hotel_id UUID REFERENCES competitor_hotels(id) ON DELETE SET NULL,
    trigger_type VARCHAR(50) NOT NULL,
    trigger_data JSONB NOT NULL DEFAULT '{}', -- 触发时的详细数据
    message TEXT NOT NULL,
    is_read BOOLEAN DEFAULT false,
    notification_status VARCHAR(20) DEFAULT 'pending', -- pending, sent, failed
    created_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_alert_records_user ON alert_records(user_id, created_at DESC);
```

# 软件设计方案

## 项目概述
本项目旨在为酒店运营者（特别是中小型酒店或连锁酒店单店）构建一套竞品房价与周边活动的实时监控与分析系统。系统通过Chrome Extension辅助采集美团商家后台数据，结合活动信息自动采集，实现数据可视化与AI智能分析，为酒店运营经理和收益管理专员提供及时、直观的市场洞察与定价决策支持。

## 1. 需求分析
### 1.1 核心用户与场景
- **核心用户**：酒店运营经理/收益管理专员。他们决策压力大、时间碎片化，需要快速掌握市场动态以调整定价策略。
- **次要用户**：酒店业主/店长。他们关注宏观趋势与经营建议，需要定期查看分析报告。
- **关键场景**：
    1.  **应对竞品突然降价**：Extension自动采集美团商家后台竞争圈数据，系统在检测到价格超过设定阈值（如15%）波动时，通过微信等渠道立即告警，并提供详细数据与应对建议。
    2.  **提前布局大型活动期间定价**：系统自动采集并评估周边考试、展会、演出等活动的住宿需求热度，帮助用户提前制定有竞争力的房价策略。

### 1.2 核心功能模块
| 模块 | 优先级 | 核心功能点 |
| :--- | :--- | :--- |
| **竞品房价监控** | P0 | Chrome Extension采集美团商家后台数据（竞争圈、行业对标、经营报表）、竞品酒店管理、价格异常实时告警。 |
| **周边活动采集** | P0 | 考试类活动采集（教育考试网、各省考试院）、展会类活动采集（中国展会网、会展中心官网、活动行）、活动需求热度自动评估（高/中/低）。 |
| **数据可视化与分析** | P0 | 竞品房价对比仪表盘（折线图、柱状图）、周边活动日历与地图、Extension状态监控。 |
| **智能分析与报告** | P1 | AI市场分析报告（对接大模型如DeepSeek）、定价建议模拟。 |
| **通知与推送** | P0 | 微信实时/定时推送（告警、活动提醒、报告摘要、Extension离线提醒）。 |
| **系统管理** | P1 | Extension状态监控、活动采集范围配置。 |

### 1.3 非功能需求与边界
- **性能**：仪表盘加载<3秒，Extension采集上报<10秒，支持单用户监控100+竞品。
- **安全**：密码加密存储、不存储美团账号密码、Extension Token认证、RBAC、数据隔离、全程HTTPS。
- **可用性**：核心服务可用性≥99.5%，界面简洁（主要操作3步内），支持移动端适配。
- **边界**：不提供自动定价与PMS同步；不替代人工决策；不代登录商家后台；MVP仅覆盖美团平台；活动采集仅采集公开数据。

## 2. 技术架构
### 2.1 整体架构
采用前后端分离架构，分为展示层、应用层、数据层和基础设施层。
```
用户浏览器
├── 前端Web应用 (Vue 3)
└── Chrome Extension → 美团商家后台数据提取 → HTTPS上报后端
                                    |
                                    v
                            后端API服务 (FastAPI)
                                    |
        +---------------------------+---------------------------+
        |                           |                           |
        v                           v                           v
[核心业务逻辑服务]          [异步任务引擎 (Celery)]      [AI分析服务]
  - 用户/竞品/活动管理       - 活动采集Worker             - 集成大模型API
  - Extension管理            - 告警检测Worker              - 报告生成
  - 告警规则处理             - 通知推送Worker
        |                           |
        v                           v
[数据库 (PostgreSQL + TimescaleDB)]  [缓存/消息队列 (Redis)]
  - 用户、配置、元数据               - 任务队列、状态缓存
  - 房价时序数据 (TimescaleDB)       - Extension在线状态
  - 活动数据、GIS数据
```

### 2.2 技术选型
| 层面 | 技术方案 | 选型理由 |
| :--- | :--- | :--- |
| **前端** | Vue 3 + TypeScript + Element Plus + ECharts | 生态成熟，类型安全，开发高效，图表能力强。 |
| **浏览器扩展** | Chrome Extension (Manifest V3) | 合规采集美团商家后台数据，用户自行登录，不存储密码。 |
| **后端** | Python (FastAPI) + Celery + Redis | Python在数据分析与AI集成方面生态强大，FastAPI高性能，Celery管理异步任务。 |
| **数据库** | PostgreSQL + TimescaleDB + PostGIS | PostgreSQL功能全面，TimescaleDB高效存储时序数据，PostGIS支持地理查询。 |
| **部署** | Docker + Docker Compose | 容器化保障环境一致性，Docker Compose适合MVP快速部署。 |

### 2.3 核心数据模型
- **竞品酒店 (`competitor_hotels`)**：关联用户，存储平台、平台ID、监控房型等。
- **房价时序数据 (`price_time_series`)**：TimescaleDB存储，包含时间戳、竞品ID、房型、价格、房态、数据来源（extension/manual）。
- **周边活动 (`surrounding_activities`)**：存储活动标题、时间、地点（GIS）、来源、类型（exam/exhibition/concert）、需求热度等。
- **Extension设备 (`extension_devices`)**：管理Extension与用户的绑定关系、在线状态、采集统计。
- **告警规则 (`alert_rules`)**：存储用户设定的价格波动阈值、关联竞品、通知方式等。

## 3. 实现方案
### 3.1 Chrome Extension实现
- **Manifest V3**：采用最新Chrome扩展标准，安全性与性能更优。
- **Content Script**：注入美团商家后台页面，自动检测页面类型并提取竞争圈、行业对标、经营报表数据。
- **选择器远程更新**：页面选择器配置存储于后端，Extension定期拉取更新，美团改版时无需用户手动升级。
- **数据上报**：通过HTTPS API将采集数据上报至后端，失败时本地暂存并重试。
- **安全设计**：不存储美团账号密码，Extension Token认证，最小权限原则。

### 3.2 后端实现
- **项目结构**：分层模块化设计，包含API路由层、核心业务服务层、数据模型层、异步任务模块、活动采集模块及AI集成模块。
- **关键接口**：
    - **Extension接口**：`POST /extension/register`（绑定）、`POST /extension/report`（数据上报）、`GET /extension/status`（状态查询）
    - **业务接口**：`GET /dashboard/price-overview`（仪表盘）、`POST /competitors`（创建竞品）、`GET /competitors/{id}/price-history`（历史价格）、`POST /alert-rules`（告警规则）、`GET /activities/calendar`（活动日历）、`POST /ai/generate-weekly-report`（AI周报）
- **异步任务**：
    - **活动采集**：Celery Beat定时触发，Worker从公开网站采集考试/展会信息。
    - **告警检测**：Extension上报数据后立即触发，比对规则后触发通知。
    - **报告生成**：AI分析服务聚合数据，调用大模型API生成报告。

### 3.3 前端实现
- **项目结构**：基于Vue 3的单页应用，组合式API，Pinia状态管理。
- **核心页面**：仪表盘、竞品管理、活动中心、告警管理、AI报告、Extension管理、系统设置。
- **关键组件**：PriceTrendChart（价格趋势图）、ActivityCalendar（活动日历）、ExtensionStatusCard（Extension状态卡片）、AlertRuleForm（告警规则表单）。

## 4. 测试策略
- **测试范围**：覆盖所有P0/P1功能，重点验证核心用户场景的端到端流程。
- **功能测试**：Extension安装绑定、数据采集上报、竞品管理、活动日历、告警触发、AI报告生成。
- **接口测试**：使用pytest对所有关键API进行自动化测试。
- **性能测试**：JMeter模拟并发，测试仪表盘加载、Extension并发上报等。
- **质量保障**：代码规范、CI/CD流水线、监控告警。

## 6. 风险与应对
| 风险 | 影响等级 | 应对方案 |
| :--- | :--- | :--- |
| **美团商家后台页面结构变更** | 高 | 1. 选择器配置化，存储于后端，支持远程更新。<br>2. 解析失败时记录日志并通知用户。<br>3. 监控采集成功率，异常时及时告警。 |
| **Extension用户使用习惯** | 中 | 1. 安装即用，无需复杂配置。<br>2. 自动采集，无需手动触发。<br>3. 长时间未采集时微信提醒用户。 |
| **活动源网站反爬或结构变更** | 中 | 1. 优先采集政府官网等反爬弱的网站。<br>2. 考试类数据更新频率低，维护成本低。<br>3. 设计降级方案，采集失败时使用缓存数据。 |
| **大模型API成本与稳定性** | 中 | 1. 报告结果缓存，避免重复调用。<br>2. 优化Prompt减少Token消耗。<br>3. 准备规则化报告模板作为降级方案。 |
| **时序数据量增长过快** | 中 | 1. TimescaleDB自动分区与压缩。<br>2. 制定数据保留策略，定期归档历史数据。 |
| **实时告警的及时性** | 高 | 1. Extension上报后立即触发告警检测。<br>2. 告警检测采用异步流式处理。<br>3. 监控微信推送送达率。 |
| **多用户数据隔离与安全** | 高 | 1. 所有数据查询强制过滤`user_id`。<br>2. Extension Token验证归属关系。<br>3. 严格RBAC，全程HTTPS。 |
